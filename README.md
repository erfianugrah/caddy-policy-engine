# caddy-policy-engine

A Caddy HTTP middleware that serves as a complete WAF — evaluating allow, block, challenge, skip, rate limit, detect (anomaly scoring), and response header rules through a 7-pass priority pipeline. Includes SHA-256 hashcash proof-of-work challenges with 6-layer bot scoring, JA4 TLS fingerprinting, session tracking, and sliding window rate limiting.

Rules are loaded from a JSON file generated by [wafctl](https://github.com/erfianugrah/wafctl) and hot-reloaded via mtime polling.

**Module ID:** `http.handlers.policy_engine`
**Listener Module ID:** `caddy.listeners.ja4`

## Features

- **7 rule types:** `allow`, `block`, `challenge`, `skip`, `rate_limit`, `detect` (anomaly scoring), `response_header`
- **7-pass evaluation pipeline** — Allow(50-99) → Block(100-149) → Challenge(150-199) → Skip(200-299) → Rate Limit(300-399) → Detect(400-499) → Response Header(500-599)
- **34 request/response fields** including multi-value aggregate fields (`all_args`, `all_headers`, `all_cookies`, `request_combined`), response fields (`response_status`, `response_body`), CRS fields (`request_line`, `request_basename`), body JSON dot-path and form extraction, plus `ja4` and `challenge_history`
- **27 operators** including regex (RE2), IP/CIDR matching, set membership, managed list membership (`in_list`/`phrase_match`), numeric comparison (`gt`/`ge`/`lt`/`le`), byte range validation, URL encoding validation, and libinjection-based `detect_sqli`/`detect_xss`
- **Challenge system** — SHA-256 hashcash proof-of-work with adaptive difficulty, 6-layer bot scoring, HMAC-signed cookies, JA4 token binding, timing validation
- **Session tracking** — Service Worker + page-level behavioral collector, beacon endpoint, JTI denylist for retrospective cookie invalidation
- **JA4 TLS fingerprinting** — `caddy.ListenerWrapper` module with hand-rolled ClientHello parser, zero external deps, full FoxIO JA4 spec
- **Sliding window rate limiting** with 16-shard concurrent counters, configurable key dimensions (including `challenge_cookie`), and jittered `Retry-After`
- **Hot-reload** via mtime polling — rules update without restarting Caddy; rate limit counters are preserved when window/limit are unchanged
- **Lazy body parsing** — JSON and form bodies are parsed at most once per request
- **CRS-compatible transforms** — 21 transform functions matching ModSecurity `t:xxx` semantics
- **Response-phase (outbound) rules** — detect rules can inspect response headers and body
- **CSP, security headers, and CORS** — per-service response header management with presets
- **Production hardening** — `hide_headers` suppresses debug response headers, regex patterns capped at 128 KiB, rule types validated at compile time

## Installation

Build a custom Caddy binary with `xcaddy`:

```bash
xcaddy build --with github.com/erfianugrah/caddy-policy-engine
```

Requires Go 1.26+ and Caddy v2.11+.

## Configuration

### Caddyfile

```caddyfile
{
    servers {
        listener_wrappers {
            ja4
            tls
        }
    }
}

:443 {
    policy_engine {
        rules_file /etc/caddy/policy-rules.json
        reload_interval 5s
        body_max_size 13mb
        hide_headers
    }
}
```

| Directive | Argument | Default | Description |
|---|---|---|---|
| `rules_file` | `<path>` | — | Path to JSON rules file |
| `reload_interval` | `<duration>` | `5s` | How often to check for file changes |
| `body_max_size` | `<size>` | `13mb` | Max body bytes to buffer (`kb`, `mb`, `gb` suffixes) |
| `hide_headers` | — | `false` | Suppress debug response headers |
| `rule` | block | — | Define an inline rule |

## Challenge System

### Overview

The challenge system serves a proof-of-work interstitial to suspicious traffic. Clients must solve a SHA-256 hashcash puzzle and pass 6-layer bot scoring to receive an HMAC-signed cookie that bypasses future challenges.

### Challenge Rule Config

| Field | Type | Default | Description |
|---|---|---|---|
| `difficulty` | int | 4 | Static PoW difficulty (leading hex zeros, 1-16). Used when adaptive range is inactive. |
| `min_difficulty` | int | 0 | Adaptive minimum difficulty (1-16, 0=disabled) |
| `max_difficulty` | int | 0 | Adaptive maximum difficulty (1-16, 0=disabled) |
| `algorithm` | string | `fast` | `fast` (normal hashing) or `slow` (10ms delay per iteration) |
| `ttl_seconds` | int | 3600 | Cookie lifetime before re-challenge |
| `bind_ip` | bool | true | Invalidate cookie if client IP changes |
| `bind_ja4` | bool | true | Invalidate cookie if JA4 TLS fingerprint changes |

**Proxy-aware bindings:** When a request arrives via a trusted proxy (detected by comparing
the trusted-proxy-resolved client IP with `r.RemoteAddr`), JA4 binding is automatically
skipped because the JA4 fingerprint belongs to the proxy's TLS stack, not the client's.
IP binding uses the resolved real client IP in all cases.

When both `min_difficulty` and `max_difficulty` are > 0, the server picks difficulty per-request based on `preSignalScore()`. Clean browsers get min, suspicious TLS/headers get max. This overrides `difficulty` entirely.

### Adaptive Difficulty

The server runs a pre-signal score at interstitial-serve time using L1 (JA4 TLS) + L2 (HTTP headers) + partial L5 (spatial) checks — no JavaScript required. The score maps linearly to the [min, max] difficulty range. Score 0 gets minimum difficulty; score >= 70 gets maximum.

### 6-Layer Bot Scoring

When the client submits the PoW solution, the server scores the submission across 6 layers:

| Layer | Source | What It Checks |
|---|---|---|
| L1 | JA4 TLS fingerprint | Non-browser ALPN (no h2), TLS 1.2-only stacks |
| L2 | HTTP headers | Missing Sec-Fetch-*, Accept-Language, Client Hints inconsistency |
| L3 | JS environment (13 probes) | webdriver, ChromeDriver markers, WebGL renderer + max texture, audio fingerprint, speech voices, permissions timing, languages, chrome.runtime, platform, memory, touch, screen |
| L4 | Behavioral (5 signals) | Mouse/key/scroll events, first interaction timing, worker timing variance |
| L5 | Spatial inconsistency | Mobile UA vs touch/screen, platform vs UA cross-check, Chrome UA vs JA4 fingerprint |
| L6 | Timing validation | Solve time vs expected minimum based on difficulty and core count |

A bot score >= 70 rejects the submission even with a valid PoW solution.

### Challenge Endpoints

| Endpoint | Method | Description |
|---|---|---|
| `/.well-known/policy-challenge/verify` | POST | PoW solution verification (returns 200 JSON `{redirect: url}`) |
| `/.well-known/policy-challenge/worker.js` | GET | Web Worker for parallel SHA-256 hashing |
| `/.well-known/policy-challenge/session-sw.js` | GET | Service Worker for session tracking |
| `/.well-known/policy-challenge/session` | POST | Session beacon receiver |

### Fail Reasons

When a challenge fails, `policy_engine.challenge_fail_reason` is set to one of:

| Reason | Description |
|---|---|
| `missing_fields` | Required fields absent from submission |
| `bad_input` | Malformed data (bad base64, JSON, etc.) |
| `payload_expired` | Challenge payload timestamp expired |
| `hmac_invalid` | HMAC signature verification failed |
| `bad_pow` | Hash doesn't meet difficulty requirement |
| `timing_hard` | Solve time impossibly fast (hard reject) |
| `bot_score` | Bot score >= 70 |

## Session Tracking

After a client passes a challenge, the plugin enables continuous behavioral monitoring:

- **Service Worker** (`session-sw.js`): Registered with origin-wide scope, intercepts navigation requests, and sends beacons with page transition data
- **Page-level collector** (`session-collector.js`): Injected into HTML responses via a wrapping `ResponseWriter` (`sessionCollectorWriter`). Collects mouse, keyboard, scroll, and timing signals, sends beacons via `navigator.sendBeacon()`
- **Beacon endpoint** (`/.well-known/policy-challenge/session`): Receives POST beacons, extracts JTI from challenge cookie for session correlation, logs via Caddy variables
- **JTI denylist** (`jti-denylist.json`): Written by wafctl, polled by the plugin. Enables retrospective invalidation of specific challenge cookies by their unique token ID

## JA4 TLS Fingerprinting

### Overview

The `caddy.listeners.ja4` module captures JA4 TLS fingerprints at the connection level, before TLS termination. Available as the `ja4` condition field in policy rules and as the `policy_engine.ja4` Caddy variable.

### Caddyfile

Place `ja4` before `tls` in the listener wrapper chain:

```caddyfile
{
    servers {
        listener_wrappers {
            ja4
            tls
        }
    }
}
```

### How It Works

1. The listener wrapper intercepts new TCP connections before TLS
2. Reads the initial bytes to capture the ClientHello message
3. Parses cipher suites, extensions, SNI, ALPN, signature algorithms via a hand-rolled binary parser (zero external dependencies)
4. Computes the full JA4 fingerprint per the FoxIO specification
5. Stores the fingerprint in a global `sync.Map` keyed by remote address
6. Replays buffered bytes via `rewindConn` so TLS handshake proceeds normally
7. Cleans up on connection close

### Use in Rules

```json
{
  "field": "ja4",
  "operator": "begins_with",
  "value": "t13d"
}
```

The `ja4` field is also available in challenge rules for cookie binding (`bind_ja4`) and in the `challenge_cookie` rate limit key.

## Rate Limiting

### Per-Rule Config

| Field | Type | Default | Description |
|---|---|---|---|
| `key` | string | `client_ip` | Bucket key (see below) |
| `events` | int | — | Max events per window |
| `window` | string | — | Duration: `1s`, `5m`, `1h`, or Go duration format |
| `action` | string | `deny` | `deny` (returns 429) or `log_only` (monitor mode) |

### Supported Keys

| Key | Description |
|---|---|
| `client_ip` | Client IP address (default) |
| `path` | URL path |
| `static` | Single global bucket |
| `client_ip+path` | Combined IP and path |
| `client_ip+method` | Combined IP and method |
| `challenge_cookie` | Challenge cookie JTI (falls back to client IP if no cookie) |
| `header:<Name>` | Named request header |
| `cookie:<Name>` | Named cookie |
| `body_json:<dotPath>` | JSON body dot-path |
| `body_form:<field>` | Form body field |

### Algorithm

Fixed-window interpolation (sliding window):

```
effectiveCount = prevCount * (1 - elapsed/window) + currCount
```

Counters are sharded across 16 independent mutexes for high-concurrency throughput. Expired counters are swept periodically (default every 30s).

### Global Config

| Field | Type | Default | Description |
|---|---|---|---|
| `sweep_interval` | string | `30s` | How often to evict expired counters |
| `jitter` | float | `0.0` | Randomization factor (0.0-1.0) for `Retry-After` header |

## Response Headers

When a rule matches, the following headers are set (unless `hide_headers` is enabled):

**Block (403):**
- `X-Blocked-By: policy-engine`
- `X-Blocked-Rule: <rule_name>`
- `X-Policy-Tags: <tags>`

**Rate Limit (under limit):**
- `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`
- `X-RateLimit-Policy: <limit>;w=<window>;name="<rule_name>"`

**Rate Limit (exceeded, 429):**
- All of the above plus `Retry-After: <seconds>` (always sent, even with `hide_headers`)

**Rate Limit (log\_only):**
- `X-RateLimit-Monitor: <rule_name>`

## Hot-Reload Behavior

- The rules file is checked every `reload_interval` (default 5s) via mtime comparison
- On change: rules are recompiled and atomically swapped under a write lock
- Rate limit counters are **preserved** when a rule's `events`/`window` stay the same
- If the file is deleted, rules are cleared (all requests pass through)
- If the initial file doesn't exist at startup, the plugin starts with empty rules and picks up the file when it appears
- The JTI denylist file is also polled and hot-reloaded

## Security

v0.20.0 includes a comprehensive security audit (March 2026) with 34 fixes:
- JSON recursion depth limit (64) and value extraction cap (10,000)
- HTML entity decode: semicolon-less numeric entities, 16 security-critical named entities
- CORS: null origin rejection, credentials+wildcard validation, auto-anchored regex, origin normalization
- CSP: RawDirectives validated for injection, mode validated at compile time
- Rate limiting: empty key falls back to client IP, max keys per zone (100K default)
- Aho-Corasick: zero-allocation inline ASCII lowering
- Transforms: jsDecode octal escapes, cmdLine backtick stripping, ES6 \u{} support
- Security headers: re-asserted in WriteHeader, strict/moderate presets
- Allow rules no longer bypass rate limits
- Spoofable allow/skip rule warnings at compile time
